# v1.0.14 Candidate Queue Implementation Skeleton

Status: implementation_ready

## Goal

Turn the candidate queue design note into an implementation-ready skeleton for the first real
resident app UI shell.

This document assumes ADR 0010 and the candidate queue design note are already accepted.

## Primary payload

Use:

```text
GET /app/screen-state/candidates
```

The queue screen should render directly from `resident_app_candidate_queue_screen_state`.

## Expected payload sections

- `reviewable_count`
- `status_counts`
- `top_sections`
- `items`
- `default_sort`

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. Queue screen loader

Responsibility:

- fetch `GET /app/screen-state/candidates`
- own loading/error/retry behavior

Existing references:

- `extension/src/screen-state-reader.ts`
- `extension/src/bridge-client.ts`

### 2. Queue screen state types

Responsibility:

- reuse the existing `CandidateQueueScreenState` shape
- avoid introducing a second queue-specific view model if the existing shape is sufficient

Existing references:

- `extension/src/types.ts`
- `src/sayane/app/app_screen_states.py`
- `src/sayane/app/app_candidate_views.py`

### 3. Queue summary renderer

Responsibility:

- render `reviewable_count`
- render `default_sort`
- render `status_counts` in stable order
- render active review context when present

### 4. Top sections renderer

Responsibility:

- render `top_sections`
- hide the section summary when no section summary exists

### 5. Queue item list

Responsibility:

- render `items`
- preserve candidate detail navigation
- keep active review candidate visually obvious when present
- avoid re-deriving review semantics in the UI shell

### 6. Empty/error state handling

Responsibility:

- show explicit loading state
- show auth failure state
- show transport failure state
- show bounded empty state when no reviewable candidates exist

## Acceptance criteria

- the queue screen renders from `GET /app/screen-state/candidates`
- the app does not need bootstrap HTML to show the queue surface
- queue summary is readable without opening candidate detail
- queue items navigate toward detail flow
- loading/auth/transport/empty states are visible and bounded
- active review context remains visible when a review session exists

## Test plan

### Contract-level

- verify the UI reads `resident_app_candidate_queue_screen_state`
- verify queue summary fields are consumed in stable order

### UI behavior

- loading state renders before fetch completion
- auth failure state renders on auth error
- transport failure state renders on fetch failure
- empty state renders when no reviewable candidates are present

### Navigation

- queue item routes to candidate detail
- active review shortcut routes to current candidate detail when present

## Non-goals

- final design system
- production auth redesign
- detail/diff/action implementation in the same issue
- direct profile patch UI

## Recommended execution order

1. wire queue screen loader
2. render queue summary
3. render top sections
4. render queue items
5. add loading/error/empty states
6. preserve active review context
7. add focused UI tests
