# v1.0.14 Production Auth Model Design Note

Status: implemented

## Goal

Define the production auth/session direction for the first real resident app UI shell under ADR
0010.

This note turns Issue 9 into a concrete implementation start point.

## Implementation outcome

Issue 9 is now closed on the app-first resident shell path.

Delivered in the current slice:

- `GET /app/ui` now treats the Bridge bearer as a bootstrap credential that can mint or replace a
  dedicated local UI session
- follow-up HTML, `/app/ui-state/*`, and `/app/ui-action/*` requests now require the dedicated UI
  session instead of accepting the raw bearer as a steady-state browser credential
- dedicated UI session replacement is covered by bootstrap re-entry
- dedicated UI session invalidation is exposed through a local logout endpoint
- focused auth/session regression coverage now pins bootstrap entry, follow-up session continuity,
  replacement, and invalidation behavior

## Decision

The real resident app should replace the bootstrap-only bearer-plus-cookie convenience seam with a
dedicated local UI session model.

Recommended direction:

1. keep the current Bridge bearer token as a bootstrap/pairing credential
2. mint a dedicated local UI session after explicit bootstrap/auth entry
3. use that dedicated UI session for browser-driven follow-up reads and writes
4. stop treating the static Bridge bearer token as the long-lived browser navigation credential

This keeps the app-facing read/write boundaries unchanged while separating bootstrap credentialing
from steady-state UI session behavior.

## Why this path

The current implementation is intentionally transitional:

- initial `GET /app/ui` requires bearer auth
- follow-up HTML navigation and form posts continue through a UI cookie
- the same static Bridge token can currently satisfy both bootstrap and follow-up session needs

That seam is useful for the bootstrap phase, but it is not a good final model because it mixes:

- bootstrap entry
- browser session continuity
- long-lived Bridge credential handling

## Session model direction

### Bootstrap credential

- the existing Bridge bearer remains the explicit bootstrap/pairing credential
- it should not become the normal long-lived browser session token

### Local UI session

- the real Sayane Resident App UI shell should use a dedicated local UI session
- that session should be scoped to the UI surface
- that session should be revocable/replaceable independently of the static Bridge bearer

### Browser continuity

- follow-up browser navigation may remain cookie-backed
- but the cookie should represent the dedicated UI session, not the raw bootstrap bearer convenience

## Boundary

This production auth slice:

- does not widen mutation scope
- does not change app-facing review/daemon endpoint semantics
- does not imply OS keychain, OAuth, or remote identity integration in the same issue
- must keep daemon wording preview/observation only
- should keep auth separate from later authorization policy work

## Required behaviors

The production auth model should provide:

- explicit bootstrap entry
- explicit steady-state UI session behavior
- session invalidation / replacement behavior
- visible auth failure handling in the UI shell
- no dependency on the bootstrap-only cookie convenience seam

## Implementation shape

Recommended implementation order:

1. define the dedicated local UI session artifact and lifecycle
2. separate bootstrap bearer verification from UI session issuance
3. update UI reads/writes to rely on the dedicated session path
4. preserve current app-facing endpoint boundaries while changing auth wiring only
5. add focused auth/session regression checks

## Non-goals

- final desktop packaging model
- OS keychain integration
- remote account model
- daemon authorization redesign
