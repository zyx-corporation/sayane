# v1.0.14 Extension and App Surface Alignment Design Note

Status: implemented

## Goal

Define how the extension compatibility surface and the resident app should align during the
production phase.

This note turns Issue 11 into a concrete implementation start point.

## Decision

The resident app should remain the primary growth surface.

The extension should remain a compatibility surface and should share app-facing contracts and
screen-state semantics where reuse reduces accidental drift, while keeping any deliberate
surface-specific behavior explicit.

## Implemented resolution

The resident app contract now makes the alignment line explicit through:

- `surface_roles`
  - `resident_app` -> `primary_growth_surface`
  - `extension` -> `compatibility_surface`
- `shared_semantics`
  - shared by design: app-facing endpoint contracts, screen-state payload semantics, review/daemon boundary wording
  - deliberately separate by design: host-container UX, auth/session handling

This keeps the app-first roadmap explicit while still preventing accidental contract drift for the
extension compatibility path.

## Why this path

ADR 0010 already chose the resident app as the primary new UI shell.

The remaining production question is not which surface wins, but how to prevent:

- accidental duplication of screen-state interpretation
- accidental auth divergence
- accidental review-semantics drift between extension and app

This issue should document which parts are shared by design and which remain intentionally separate.

## Alignment direction

### Shared by design

- app-facing endpoint contracts
- screen-state payload semantics
- core review and daemon wording boundaries

### Potentially separate by design

- host container UX
- launch ergonomics
- compatibility-only affordances in the extension

## Boundary

This issue:

- does not make the extension the primary new shell
- does not fork candidate or daemon semantics
- does not require identical visual presentation across both surfaces

## Implementation shape

Recommended implementation order:

1. document shared contracts and screen-state interpretation
2. document deliberate surface-specific behavior
3. reduce duplicate auth and review wiring where feasible
4. add drift checks where shared behavior matters
