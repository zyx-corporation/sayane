# v1.0.14 Candidate Detail and Diff Design Note

Status: implementation_ready

## Goal

Define the first real resident app candidate detail and diff implementation slice under ADR 0010.

This note turns Issue 4 into a concrete implementation start point.

## Decision

The first real resident app candidate detail screen should use:

```text
GET /app/screen-state/candidates/{id}
```

as its primary rendering payload.

The first real diff preview should continue to use:

```text
GET /app/candidates/{id}/diff
```

as a companion read path until a dedicated diff screen-state surface is needed.

`GET /app/candidates/{id}` remains the primary app-facing detail read model, but the first real
detail screen should consume the framework-neutral detail screen state builder rather than
re-deriving detail summary and action semantics in the UI shell.

## Why this path

The current app layer already provides:

- `build_app_candidate_detail(payload)`
- `build_app_candidate_diff(payload)`
- `build_candidate_detail_screen_state(detail_payload, diff_available=...)`
- `GET /app/screen-state/candidates/{id}`

This state already expresses the detail screen in UI-oriented terms:

- `ui_summary`
- `allowed_actions`
- `proposal`
- `evaluation`
- `content`
- `diff_available`

The existing extension sidepanel also proves the expected detail semantics:

- detail summary keeps active review context visible
- allowed actions are exposed as bounded hints, not ad-hoc UI inference
- diff preview is presentation-only and fetched after detail state resolves
- lineage and diff previews fail closed back to queue if the detail candidate can no longer load

Using `GET /app/screen-state/candidates/{id}` for the first real detail screen keeps the
Bridge-hosted Sayane Resident App local shell focused on rendering and navigation wiring, while keeping diff as
an explicit review aid over the existing read path.

## Screen contents

The first real candidate detail screen should render:

1. detail summary
   - candidate id
   - active review presence/source/count
   - active review note when the visible candidate is not the active candidate
   - diff availability
   - `ui_summary` fields
2. proposal payload
3. evaluation payload
4. action availability summary
5. content body
6. presentation-only diff preview when available
7. lineage preview

## Required states

The first real detail/diff screen must include:

- loading state
- auth failure state
- transport failure state
- fail-closed fallback to queue when the visible detail candidate can no longer load
- graceful handling when diff is unavailable

## Boundary

This detail/diff screen:

- does not mutate candidate/profile state directly in this issue
- does not treat diff as a patch-apply surface
- does not overclaim action readiness beyond `allowed_actions`
- keeps lineage/diff as review aids inside the existing candidate boundary

## Implementation shape

Recommended implementation order:

1. create a detail screen loader around `GET /app/screen-state/candidates/{id}`
2. render detail summary in stable field order
3. render proposal/evaluation/content sections
4. render action availability hints from `allowed_actions`
5. fetch and render diff preview only when `diff_available` is true
6. fetch and render lineage preview
7. add loading/error/fallback states

## Relationship to other surfaces

- home screen should use `GET /app/screen-state/home`
- candidate queue should use `GET /app/screen-state/candidates`
- action flow should later use the existing review mutation endpoints
- daemon panel should use `GET /app/screen-state/daemon`

This keeps the first real Sayane Resident App UI shell aligned with the existing screen-state contract while preserving
the current diff read path as a bounded companion surface.
