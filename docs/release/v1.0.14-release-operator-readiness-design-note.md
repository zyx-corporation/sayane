# v1.0.14 Release and Operator Readiness Design Note

Status: implemented

## Goal

Define the production release/readiness slice required before the resident app can be handed to an
operator without code-reading as the primary support path.

This note turns Issue 13 into a concrete implementation start point.

## Implementation outcome

Issue 13 is now closed on the app-first resident shell path.

Delivered in the current slice:

- one explicit resident-app operator handoff document now covers startup, config, logs, troubleshooting, and escalation references
- Bridge/token/UI-session expectations are now documented with the current dedicated local UI session model
- operator troubleshooting now points to concrete Bridge, overview, daemon-overview, and readiness-diagnostic checks
- contract metadata now exposes UI session logout as part of the supported local shell write surface

## Decision

The production resident app should ship with one explicit operator-readiness line covering:

- package/startup expectations
- config handling
- logs and failure inspection points
- troubleshooting and runbook guidance

This release/readiness line should stay conservative and should not widen daemon/runtime claims
beyond what the current implementation and docs can actually support.

## Why this path

By Issue 13, the remaining gap is not core screen behavior but operator handoff quality:

- how to start the app
- how to confirm expected local prerequisites
- how to interpret failures
- where to find the right runbook or policy document

The project already has release-prep and operator handoff patterns elsewhere, so Issue 13 should
reuse that discipline for the resident app rather than improvising an informal release story.

## Required deliverables

The production readiness slice should define:

1. packaging and startup expectations
2. required config/runtime assumptions
3. log and diagnostic entry points
4. troubleshooting paths for common failures
5. operator handoff/runbook references

## Boundary

This issue:

- does not silently widen daemon/runtime claims
- does not turn preview surfaces into proof claims
- does not replace explicit policy/runbook references with vague UI-only reassurance

## Implementation shape

Recommended implementation order:

1. define packaging/startup expectations for the chosen shell
2. define config and environment assumptions
3. define log/diagnostic inspection points
4. assemble operator-facing troubleshooting guidance
5. add a concise handoff/runbook line tied to existing architecture/policy docs
