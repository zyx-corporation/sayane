# v1.0.14 MVP UI State Handling and Regression Design Note

Status: implemented

## Goal

Define the MVP-wide state-handling and regression-coverage slice for the first real resident app UI
shell under ADR 0010.

This note turns Issue 8 into a concrete implementation start point.

## Implementation outcome

Issue 8 is now closed on the app-first resident shell path.

Delivered in the current slice:

- explicit loading states remain visible across home, queue, detail, and daemon loaders
- auth/session failures are rendered distinctly from generic fetch failures in the embedded shell
- transport failures are rendered distinctly from auth/session failures
- queue and daemon routes now fail in a bounded way instead of leaving the shell in a loading-only state
- detail refresh still fail-closes back to queue when the current candidate cannot be safely reloaded
- focused regression coverage now pins auth failure messaging and bounded shell fallback behavior

## Decision

The first real resident app MVP should standardize state handling across the home, queue, detail,
capture, action, and daemon surfaces using the existing screen-state/read-surface flow plus bounded
fallback behavior already present in the compatibility sidepanel.

Regression coverage should begin with focused contract/UI-path checks rather than a broad new test
harness.

## Why this path

The current implementation already proves the main state-handling seams:

- home, queue, detail, and daemon loaders each expose explicit loading text
- Bridge/API failures already map into bounded unavailable/error messages
- detail refresh already fail-closes back to queue when the current candidate cannot be reloaded
- screen-state payloads already define most of the state shape needed by the real Sayane Resident App UI shell

The current test surface also already provides useful anchors:

- focused Bridge API tests for home/queue/detail/daemon/capture/action slices
- acceptance scenario registry for existing L1 coverage tracking
- UI-side compatibility implementation that shows where loading/fallback state is currently expected

Using these existing seams for Issue 8 keeps the MVP close to current behavior instead of inventing
a separate error-handling model late in the slice.

## Required state classes

The MVP UI shell should explicitly handle:

1. loading
2. auth failure
3. transport failure
4. bounded empty state
5. operation error
6. fail-closed fallback where the current screen can no longer be safely rendered

## State policy

### Loading

- each screen loader should render a visible in-flight state before payload resolution

### Auth failure

- auth/session problems should render explicitly
- the UI should not silently collapse auth failure into generic empty state

### Transport failure

- network/Bridge unavailability should render explicitly
- the UI should preserve enough navigation context for retry or fallback

### Empty state

- empty state should render only when the underlying payload is valid but has no meaningful items
- empty state must not imply failure when the result is expected

### Operation error

- mutation failures should render bounded action feedback
- screen context should remain visible when safe

### Fallback

- when detail refresh cannot reload the visible candidate, the UI should fail closed back to queue
- similar fallback behavior should prefer a safe higher-level surface over stale data

## Regression coverage direction

The first MVP regression pass should cover:

- one primary home load path
- one queue navigation path
- one detail + diff path
- one action-flow mutation path
- one clipboard capture path
- one daemon panel path
- one fail-closed detail fallback path

## Boundary

This Issue 8 slice:

- does not require a full production E2E harness
- does not redesign auth/session architecture
- does not widen daemon/candidate mutation boundaries
- must keep wording aligned with the current i18n direction instead of hard-coding final-only copy

## Implementation shape

Recommended implementation order:

1. define shared state-handling expectations for each screen
2. align home/queue/detail/daemon loaders with the same loading/error conventions
3. preserve fail-closed queue fallback for detail refresh failure
4. add focused regression checks for the primary MVP path set
5. extend acceptance coverage mapping only where it helps track the new MVP path set
