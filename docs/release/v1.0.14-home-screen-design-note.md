# v1.0.14 Home Screen Design Note

Status: implementation_ready

## Goal

Define the first real resident app home screen implementation slice under ADR 0010.

This note turns Issue 2 into a concrete implementation start point.

## Decision

The first real resident app home screen should use:

```text
GET /app/screen-state/home
```

as its primary rendering payload.

`GET /app/overview` remains the primary app-facing read model for the overall resident app boundary,
but the first real home screen implementation should consume the framework-neutral home screen state
builder rather than re-deriving home view state in the UI shell.

## Why this path

The current app layer already provides:

- `build_home_screen_state(overview)`
- `GET /app/screen-state/home`

This state already expresses the home screen in UI-oriented terms:

- `summary_cards`
- `top_review_items`
- `top_daemon_actions`
- `quick_links`

Using `GET /app/screen-state/home` for the first real home screen keeps the Bridge-hosted local web
app shell focused on rendering and interaction wiring rather than duplicating view-model logic.

## Screen contents

The first real home screen should render:

1. summary cards
   - repository availability
   - reviewable count
   - approved context count
   - blocked context count
   - daemon state
   - next action count
2. top review items
3. top daemon actions
4. quick links

## Required states

The first real home screen must include:

- loading state
- auth failure state
- transport failure state
- empty-state handling when repository-backed review data is absent

## Boundary

This home screen:

- does not add mutation capability
- does not patch profile state directly
- does not infer daemon proof/readiness beyond the current screen-state payload
- does not replace the underlying `GET /app/overview` contract

## Implementation shape

Recommended implementation order:

1. create a home screen loader around `GET /app/screen-state/home`
2. render summary cards with stable card ordering
3. render top review items with detail navigation hooks
4. render top daemon actions with daemon-panel navigation hooks
5. render quick links
6. add loading/error/empty states

## Relationship to other surfaces

- candidate queue should later use `GET /app/screen-state/candidates`
- candidate detail should later use `GET /app/screen-state/candidates/{id}`
- daemon panel should later use `GET /app/screen-state/daemon`

This keeps the first real Sayane Resident App UI shell aligned with the existing screen-state contract instead of mixing
raw payload composition and derived view state arbitrarily.
