# v1.0.14 Candidate Action Flow Implementation Skeleton

Status: implementation_ready

## Goal

Turn the candidate action-flow design note into an implementation-ready skeleton for the first real
resident app UI shell.

This document assumes ADR 0010 and the candidate action-flow design note are already accepted.

## Primary write surfaces

Use:

```text
POST /app/candidates/{id}/evaluate
POST /app/candidates/{id}/revise
POST /app/candidates/{id}/approve
POST /app/candidates/{id}/reject
```

The action launcher should start from `resident_app_candidate_detail_screen_state`.

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. Action submit client

Responsibility:

- send evaluate/revise/approve/reject requests
- normalize auth/transport/operation errors

Existing references:

- `extension/src/bridge-client.ts`
- `src/sayane/bridge/routes/resident_app.py`

### 2. Detail action binding

Responsibility:

- bind available actions from `CandidateDetailScreenState.allowed_actions`
- disable or hide unavailable actions without inventing local policy

Existing references:

- `extension/src/types.ts`
- `extension/src/resident-app-sidepanel.ts`
- `src/sayane/app/app_candidate_views.py`

### 3. Success navigation and refresh

Responsibility:

- refresh detail after evaluate
- route revised candidates into the new candidate detail
- refresh queue/detail after approve or reject

### 4. Critical approval and confirmation UI

Responsibility:

- support `force_critical`
- support `override_reason`
- support section-specific `explicit_confirmation`
- preserve server-side validation as the source of truth

### 5. Localized feedback

Responsibility:

- surface success and error feedback through the current resident-app translation path
- keep message keys and wording compatible with the broader i18n direction

### 6. Loading/error state handling

Responsibility:

- show action-in-flight state
- show auth failure state
- show transport failure state
- show structured operation errors from the existing Bridge payloads

## Acceptance criteria

- the real Sayane Resident App UI shell can evaluate, revise, approve, and reject through the existing app-facing endpoints
- action availability comes from `allowed_actions`
- revise creates and routes to a new pending candidate
- approve preserves critical/explicit-confirmation safeguards
- loading/auth/transport/operation-error states are visible and bounded
- localized feedback remains compatible with the resident-app i18n path

## Test plan

### Contract-level

- verify the UI does not offer unavailable actions when `allowed_actions` disables them
- verify mutation calls use the existing app-facing write surfaces

### UI behavior

- evaluate refreshes the candidate detail state
- revise routes to the new candidate detail
- approve handles critical-approval failure and explicit-confirmation failure cleanly
- reject refreshes the visible review state

### Error handling

- auth failure renders explicitly
- transport failure renders explicitly
- operation error payloads render bounded action feedback without losing navigation context

## Non-goals

- final design system
- production auth redesign
- queue/detail screen implementation in the same issue
- direct profile patch UI

## Recommended execution order

1. wire action submit client
2. bind available actions from detail state
3. implement evaluate and refresh
4. implement revise and reroute
5. implement approve/reject with server-driven error handling
6. add localized feedback states
7. add focused UI tests
