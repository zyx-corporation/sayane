# v1.0.14 Resident App Bootstrap UI Handoff

Status: handoff_ready

## Summary

The current resident app phase now includes a local Bridge HTML bootstrap UI for the 紗綾音 Resident App local shell.

This is not the final GUI framework.

It is a local presentation seam that proves the current app-facing resident surfaces can be
navigated end-to-end by a human without inventing new mutation or daemon boundaries.

## Implemented local HTML surfaces

The bootstrap UI currently provides:

```text
GET /app/ui
GET /app/ui/candidates
GET /app/ui/candidates/{id}
GET /app/ui/candidates/{id}/diff
GET /app/ui/daemon
```

It also provides local HTML form actions for:

```text
POST /app/ui/capture-clipboard
POST /app/ui/candidates/{id}/evaluate
POST /app/ui/candidates/{id}/approve
POST /app/ui/candidates/{id}/reject
POST /app/ui/candidates/{id}/revise
```

The local shell wording is now aligned in both English and Japanese for:

- queue / detail / diff / daemon shell headings
- review status labels such as pending / evaluated
- RDE class display fallbacks
- read-surface purpose labels
- action success notices and review-transition error feedback
- extension diff viewer headings and action labels
- embedded shell raw JSON panels now localize value-side status / phrase tokens before display
- daemon panel preview tables now localize cleanup / repair value tokens such as directory roles and no-action recommendations
- operator-workspace framing, session actions, skip-link copy, and shell-workspace labels are now aligned in both locales
- shell layout now uses shared focus-visible, summary-first panel rhythm, and narrow-width responsive behavior

## What this bootstrap UI proves

- contract / overview payloads are enough to drive a human-facing home screen
- candidate queue/detail/diff payloads are enough to drive a human-facing review flow
- clipboard capture can enter the same local review flow
- daemon overview preview is enough to drive a bounded daemon panel
- redirect-based success/error feedback can be layered on top without changing the app-facing API boundary

## What it explicitly does not prove

- final GUI framework choice
- production auth model
- production daemon identity proof
- production daemon readiness proof
- production API readiness proof
- direct profile patch UI
- OS service integration UI

## Current interaction model

The bootstrap UI currently uses:

1. bearer token on the initial `GET /app/ui` as the bootstrap or pairing credential
2. dedicated local UI session cookie for follow-up HTML navigation, local JSON reads, and local JSON writes
3. redirect-based success notices
4. redirect-based validation / transition error feedback

This bootstrap seam now mints a dedicated local UI session for the real resident shell.

The raw Bridge bearer is no longer the steady-state browser session credential for follow-up local
UI activity.

## Recommended next work

1. Keep `GET /app/overview` as the primary read model.
2. Keep candidate mutation inside the existing review endpoints.
3. Preserve daemon panel wording as preview/observation only.
4. Decide whether the next phase is:
   - richer bootstrap HTML refinement, or
   - final GUI implementation against the same app-facing endpoints.
