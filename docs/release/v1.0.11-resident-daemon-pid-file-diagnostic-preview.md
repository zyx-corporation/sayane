# Sayane v1.0.11 Resident Daemon PID File Diagnostic Preview

Release date: 2026-06-17

## Summary

Sayane v1.0.11 packages the resident daemon PID file diagnostic preview track.

This release adds a read-only PID file parse diagnostic for the planned resident daemon PID artifact. The diagnostic classifies PID file content without proving daemon liveness, probing processes, sending signals, controlling processes, or mutating filesystem state.

The resident daemon preview sequence is now:

```text
identity preview -> lifecycle preview -> operation plan preview -> runtime layout preview -> stale artifact diagnostic preview -> cleanup decision preview -> PID file parse diagnostic preview
```

## Added commands

PID file diagnostic preview:

```bash
sayane app daemon-pid-diagnostic --json
```

The command supports a runtime root override:

```bash
sayane app daemon-pid-diagnostic --runtime-root /path/to/runtime --json
```

## PID parse classifications

The diagnostic preview reports planned PID file content as:

```text
missing
unreadable
empty
invalid
parsed
```

`parsed` means only that the PID file content was read as a positive integer string. It is not a daemon liveness proof, not proof that the process exists, and not proof that any process is Sayane.

## Safety boundary

This release remains preview-only.

The new command observes and parses the planned PID file. It does not create, delete, repair, or rewrite the PID file. It does not acquire daemon locks, create IPC endpoints, probe process liveness, send signals, or control daemon processes.

The payload explicitly keeps safety indicators false:

```text
proves_liveness: false
probes_process: false
controls_process: false
mutates_filesystem: false
```

## RDE Delta-M Review

### Preserved

Resident daemon work remains staged and auditable.

Preview commands remain non-mutating.

PID file presence remains explicitly separated from daemon liveness.

### Supplemented

The system now separates PID-file parsing from stronger future process verification:

```text
file observation -> content parse -> future liveness verification
```

Only observation and parsing exist in v1.0.11.

Liveness verification and process control remain out of scope.

### Deviation Risks

Do not treat `parsed` as proof that a daemon is running.

Do not treat `parsed` as proof that a process is Sayane.

Do not add process probing, signal sending, or filesystem mutation behavior to the preview command.

Do not reuse the preview command name for a future mutating daemon-control command.
