# v1.0.14 Home Screen Implementation Skeleton

Status: implementation_ready

## Goal

Turn the home screen design note into an implementation-ready skeleton for the first real resident
app UI shell.

This document assumes ADR 0010 and the home screen design note are already accepted.

## Primary payload

Use:

```text
GET /app/screen-state/home
```

The home screen should render directly from `resident_app_home_screen_state`.

## Expected payload sections

- `summary_cards`
- `top_review_items`
- `top_daemon_actions`
- `quick_links`

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. Home screen loader

Responsibility:

- fetch `GET /app/screen-state/home`
- own loading/error/retry behavior

Existing references:

- `extension/src/screen-state-reader.ts`
- `extension/src/bridge-client.ts`

### 2. Home screen state types

Responsibility:

- reuse the existing `HomeScreenState` shape
- avoid introducing a second home-specific view model if the existing shape is sufficient

Existing references:

- `extension/src/types.ts`
- `src/sayane/app/app_screen_states.py`

### 3. Summary card renderer

Responsibility:

- render `summary_cards` in stable order
- preserve the current keys:
  - `repository`
  - `reviewable`
  - `approved_context`
  - `blocked_context`
  - `daemon_state`
  - `next_actions`

### 4. Top review items list

Responsibility:

- render `top_review_items`
- route into candidate detail flow
- avoid re-deriving review status semantics in the UI shell

### 5. Top daemon actions list

Responsibility:

- render `top_daemon_actions`
- route into daemon panel flow where appropriate
- preserve preview/observation wording

### 6. Quick links

Responsibility:

- render `quick_links`
- keep queue and daemon panel navigation obvious

## Required UI states

### Loading

- visible while `GET /app/screen-state/home` is in flight

### Auth failure

- explicit failure state when Bridge auth/session is unavailable

### Transport failure

- explicit failure state when the request fails or the Bridge is unavailable

### Empty data

- render gracefully when repository-backed review data is absent
- do not imply failure when the empty state is expected

## Acceptance criteria

- the home screen renders from `GET /app/screen-state/home`
- the app does not need bootstrap HTML to show the first real home screen
- summary cards are readable without opening deeper panels
- review items navigate toward detail flow
- daemon actions navigate toward daemon flow
- loading/auth/transport/empty states are visible and bounded

## Test plan

### Contract-level

- verify the UI reads `resident_app_home_screen_state`
- verify summary card keys are consumed in stable order

### UI behavior

- loading state renders before fetch completion
- auth failure state renders on auth error
- transport failure state renders on fetch failure
- empty state renders when repository-backed review data is absent

### Navigation

- quick link to candidate queue works
- quick link to daemon panel works
- top review item routes to candidate detail

## Non-goals

- final design system
- production auth redesign
- queue/detail/daemon implementation in the same issue
- direct profile patch UI

## Recommended execution order

1. wire home screen loader
2. render summary cards
3. render top review items
4. render top daemon actions
5. render quick links
6. add loading/error/empty states
7. add focused UI tests
