# v1.0.14 Daemon Panel Design Note

Status: implementation_ready

## Goal

Define the first real resident app daemon-panel implementation slice under ADR 0010.

This note turns Issue 7 into a concrete implementation start point.

## Decision

The first real resident app daemon panel should use:

```text
GET /app/screen-state/daemon
```

as its primary rendering payload.

`GET /app/daemon-overview` remains the primary app-facing daemon preview read model, but the first
real daemon panel should consume the framework-neutral daemon panel screen state builder rather than
re-deriving daemon summary/view logic in the UI shell.

## Why this path

The current app layer already provides:

- `build_daemon_overview_preview(...)`
- `build_daemon_panel_screen_state(daemon_payload)`
- `GET /app/screen-state/daemon`

This state already expresses the daemon panel in UI-oriented terms:

- `summary_cards`
- `next_actions`
- `runtime_init`
- `cleanup_preview`
- `repair_preview`

The current implementation already proves key daemon semantics:

- daemon data is preview/observation oriented rather than proof of production readiness
- runtime init, cleanup, and repair remain explicit bounded previews
- next actions are derived operator guidance, not automatic mutation
- the existing HTML and sidepanel surfaces already render the same preview line

Using `GET /app/screen-state/daemon` for the first real daemon panel keeps the Bridge-hosted local
web app shell focused on rendering and navigation wiring rather than duplicating daemon preview
composition.

## Screen contents

The first real daemon panel should render:

1. summary cards
   - state
   - is running daemon
   - runtime initialized
   - readiness status
2. next actions
3. runtime init preview
4. cleanup preview
5. repair preview

## Required states

The first real daemon panel must include:

- loading state
- auth failure state
- transport failure state
- bounded empty-state handling when a preview section has no items

## Boundary

This daemon panel:

- does not claim daemon identity proof or production readiness proof
- does not treat preview reachability as permission to mutate runtime state
- does not introduce direct daemon mutation UI in this issue
- must keep user-facing copy compatible with the existing preview/observation wording and i18n direction

## Implementation shape

Recommended implementation order:

1. create a daemon panel loader around `GET /app/screen-state/daemon`
2. render summary cards in stable order
3. render next actions
4. render runtime init / cleanup / repair previews
5. add loading/error/empty states

## Relationship to other surfaces

- home screen should link into the daemon panel through its quick links and top daemon actions
- queue/detail/action/capture flows remain separate from daemon observation flow
- any future daemon mutation UI must remain a separate bounded issue

This keeps the first real Sayane Resident App UI shell aligned with the current local-only daemon preview boundary.
