# v1.0.15 Resident Daemon Proof Issue Breakdown

Status: implemented_for_issue_1_2_3_4_5_6_identity_readiness_api_status_cli_wording_contracts

## Goal

Split the resident daemon proof phase into issue-sized steps that preserve current conservative
boundaries.

## Issue 1: Identity proof contract

Status:

- completed with a conservative app-layer identity-proof preview contract and CLI surface

Scope:

- define the first app-layer payload for daemon identity proof
- distinguish planned-path preview from real process ownership evidence
- keep runtime-root scoping explicit

Done when:

- one identity-proof contract exists without claiming readiness or API usability

## Issue 2: Readiness proof contract

Status:

- completed with an operation-class-aware readiness-proof preview contract and CLI surface

Scope:

- define operation-class-specific readiness proof payload
- keep liveness and readiness explicitly separate
- classify degraded, unverified, and not-ready outcomes conservatively

Done when:

- one readiness-proof contract exists without claiming API readiness

## Issue 3: API readiness proof contract

Status:

- completed with an API-readiness-proof preview contract and CLI surface

Scope:

- define API reachability/authn/authz/protocol/operation-class proof payload
- keep socket presence and endpoint reachability below proof threshold

Done when:

- one API-readiness contract exists without collapsing proof layers together

## Issue 4: Shared proof status vocabulary

Status:

- completed with a shared proof-status module and explicit downgrade-reason fields across proof contracts

Scope:

- align identity/readiness/API statuses
- make ambiguity and downgrade reasons explicit
- prevent plain generic `ready` wording

Done when:

- proof surfaces use one conservative status vocabulary

## Issue 5: CLI proof diagnostics

Status:

- completed with focused proof CLI commands plus one aggregated `daemon-proof-diagnostics` read surface

Scope:

- expose the proof contracts through CLI-oriented diagnostics
- keep invocation explicit about operation class where required
- preserve non-mutating behavior

Done when:

- operators can inspect proof-oriented payloads without mutating state

## Issue 6: Operator/app wording alignment

Status:

- completed with operator handoff, CLI reference, Bridge manual, and daemon-panel boundary wording aligned to the proof-preview line

Scope:

- update operator handoff and daemon panel wording only where proof contracts justify it
- keep preview wording where proof is still absent

Done when:

- wording matches actual evidence strength across CLI, docs, and app shell

## Recommended order

```text
1 -> 2 -> 3 -> 4 -> 5 -> 6
```

## Guardrails

- do not treat process existence as identity proof
- do not treat liveness as readiness
- do not treat socket presence as API readiness
- do not change daemon panel wording ahead of evidence
- do not mix proof work with OS service or supervision UX
