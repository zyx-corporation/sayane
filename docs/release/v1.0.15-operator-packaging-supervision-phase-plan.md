# v1.0.15 Operator Packaging and Supervision Phase Plan

Status: closed_for_mvp_native_first_release

## Goal

Record the finalized MVP operator boundary for packaging, supervision, recovery, and the native
macOS primary path, while leaving broader cross-platform packaging expansion to post-MVP work.

## Why this phase exists

The current resident app MVP is a native-first macOS workflow over existing app-facing resident
endpoints, with a Bridge-hosted browser shell retained only for debug compatibility.

It intentionally does not provide:

- OS service integration UI
- tray or background supervision UI
- a final packaging/run model beyond local CLI + Local Bridge

These are post-MVP items, not incomplete pieces of the current app shell.

## Scope

This document now records the MVP decision on how operators are expected to start, supervise,
recover, and stop the resident system, and separates any broader packaging ambition into post-MVP
work.

## Planned workstreams

### 1. Packaging model decision

Current implemented baseline:

- current aggregate read surface now provides one conservative snapshot across packaging, service, supervision, and recovery
- current operator-facing contract now records `cli_first_local_bridge` as the active packaging line
- current contract explicitly separates this active line from deferred OS service integration
- current cross-platform service target matrix now records macOS / Linux / Windows in one shared contract
- current macOS target now has concrete LaunchAgent preview/apply support

Decision now closed for MVP:

- keep `cli_first_local_bridge` as the supported packaging line
- make the native macOS app the primary operator-facing UI on macOS
- keep `/app/ui` as debug-only compatibility

Current contract baseline now records three explicit model states:

- `cli_first_local_bridge` as the current supported line
- `hybrid_local_bridge_plus_service_targets` as the next candidate if service lifecycle closes
- `service_first_resident_runtime` as a larger architecture-change candidate, not the current direction

### 2. OS service integration line

Current implemented baseline:

- current aggregate read surface now exposes current target, allowed service commands, and deferred service commands together
- current operator-facing contract now records allowed local CLI control separately from deferred service-manager commands
- current contract explicitly keeps service install/enable/disable/remove outside the current app shell
- current macOS LaunchAgent slice now writes a reviewed plist without silently activating the service
- current service target matrix now also records platform-policy gates, rollback-policy gates, and the hybrid-packaging gate explicitly
- current bridge-hosted daemon shell and native macOS runbook now both expose the same proof-oriented read commands beside this operator phase

Decision now closed for MVP:

- keep LaunchAgent preview/apply/bootstrap/bootout/kickstart as the only service-adjacent macOS path
- keep install/remove/update/enable/disable outside MVP support
- keep service lifecycle mutation CLI-only and not exposed from the app UI

Current contract baseline now also records deferred lifecycle operations explicitly:

- install
- enable
- disable
- remove
- update

### 3. Supervision UX line

Current implemented baseline:

- current aggregate read surface now exposes passive visibility, limited active supervision, and deferred background surfaces together
- current operator-facing contract now records passive local observation as supported
- current contract keeps active supervision limited to explicit CLI control
- tray, menu-bar, and background supervision surfaces remain explicitly deferred

Decision now closed for MVP:

- keep supervision passive and local-only
- allow bounded CLI daemon start/stop/restart only
- leave tray, menu-bar, and background supervision to post-MVP work

Current contract baseline now records candidate background surfaces explicitly:

- tray supervision
- menu-bar supervision
- login-item or background-agent visibility

These remain decision inputs only, not current operator commitments.

### 4. Recovery and consent line

Current implemented baseline:

- current aggregate read surface now exposes consent and recovery alignment together with packaging/supervision status
- current operator-facing contract now records diagnostics-first recovery and explicit CLI confirmation for mutating recovery steps
- current contract explicitly keeps local app recovery guidance separate from CLI consent-bearing mutations

Decision now closed for MVP:

- keep destructive or stateful recovery on explicit CLI confirmation paths
- keep app UI recovery guidance read-first
- prevent supervision or automation from bypassing review and consent boundaries

## Exit criteria

This phase is closed for MVP when it can answer all of the following:

- what the supported operator packaging model is
- what daemon control and supervision actions are allowed
- which actions remain CLI-only
- what local app UI is allowed to expose
- what recovery path operators should use when supervision fails

Current aggregate phase status now exposes a conservative closure checklist and empty
blocking-reason set so MVP closure can be tracked explicitly.

That phase-status payload is now available through both:

- `sayane app daemon-operator-phase-status --json`
- `GET /app/operator-phase-status`

The current resident app daemon shell and legacy daemon panel also surface that same operator-phase
line as a read-only detail section, including:

- current operator path
- normalized workstream status summaries
- recommended implementation order
- read surfaces
- exit criteria
- not-in-scope reminders

The same post-app line is now also exposed through human-readable CLI summaries for:

- `sayane app daemon-packaging-status`
- `sayane app daemon-service-targets-status`
- `sayane app daemon-service-control-boundary`
- `sayane app daemon-supervision-status`
- `sayane app daemon-recovery-consent-status`
- `sayane app daemon-operator-phase-status`

These text summaries are intended for operator handoff, terminal-first review, and bounded
troubleshooting. They remain read surfaces only.

## Not in scope

- daemon identity proof by itself
- daemon readiness or API readiness proof by itself
- direct profile patch UI
- replacing the current candidate review boundary

## Recommended implementation order

The MVP closure slice is complete. Further work moves to post-MVP packaging, supervision, and
cross-platform expansion tracks.
