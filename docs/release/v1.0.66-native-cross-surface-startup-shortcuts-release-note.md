# v1.0.66 Native Cross-Surface Startup Shortcuts Release Note

## Summary

- aligned the native macOS `Home`, `Bridge Status`, `Daemon`, and fallback
  `Error` surfaces around the same startup-command and debug-shell shortcuts
- exposed `Open Launcher` wherever the current startup command resolves to a
  local script path
- exposed `Open Debug Shell` and `Copy Debug Shell URL` wherever the current
  debug compatibility path is surfaced
- updated operator handoff and release index docs so the same startup guidance
  is described consistently across implementation and release references

## Why

- operator launch guidance was already present in several native surfaces, but
  the actions were uneven and still pushed users toward manual copy steps on
  some screens
- fallback and recovery screens are exactly where startup/debug shortcuts need
  to stay consistent, because those are the moments when the operator should
  not have to remember a different path
- the current resident app direction is native-macOS-first, so startup and
  debug affordances should stay recognizable before and after the user enters
  the deeper daemon workspace

## Highlights

- the native `Daemon` screen now keeps the same startup/debug actions in:
  - `Packaging Models`
  - `Startup Visibility`
  - `Supported Path`
  - `Operator Summary Rail`
- the native `Home` screen now reuses the same launch guidance through:
  - `Bridge Status`
  - `Start Here`
- the fallback native `Error` view now exposes the same launcher/debug-shell
  actions instead of only offering command copying
- operator handoff docs now explicitly call out these shared startup/debug
  shortcuts as part of the supported macOS operator path

## Boundary

- this release does not change the supported packaging model away from
  `cli_first_local_bridge`
- this release does not re-open `/app/ui` as a primary operator surface
- this release does not add new service-mutating native controls
- this release does not change the current bearer/UI-session auth model
