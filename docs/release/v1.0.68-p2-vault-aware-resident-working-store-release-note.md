# v1.0.68 P2 Vault-Aware Resident Working Store Release Note

## Summary

Sayane `v1.0.68` starts Issue `P2 Sensitive working-store migration`.

This slice moves the resident app working path for `Candidate`, `ReviewDecision`, and `Lineage`
 away from transitional FileSystem storage and onto vault-aware repositories when a Local Vault
backend is active.

The same migration line now also covers the generic legacy bridge candidate and lineage surfaces.

The current CLI `capture`, `candidate`, `review`, `context-compile`, and `audit` commands now also support the same migration line through
explicit `--vault-mode` execution.

## Included work

- `BridgeConfig` now carries optional repository and unlock-session context
- Candidate / ReviewDecision / Lineage storage facades now use repository-backed persistence when present
- resident app capture / review / revision / lineage routes now resolve vault-aware config per request
- legacy bridge `/capture`, `/candidates*`, and `/captures/*/lineage` now resolve the same vault-aware config per request
- resident app candidate queue / detail / diff / lineage reads now require an active unlock session when Local Vault is active
- resident app capture into Local Vault no longer writes `~/.sayane/candidates/<id>.json`
- legacy bridge capture into Local Vault also no longer writes `~/.sayane/candidates/<id>.json`
- CLI `capture`, `candidate`, `review`, `context-compile`, and `audit` now support explicit Local Vault execution via `--vault-mode`
- vault-backed CLI `audit` reads are derived from persisted vault-backed `ReviewDecision` records instead of a separate filesystem audit file

## Boundary retained

- transitional FileSystem working stores still exist for non-resident and legacy paths

## Follow-up closure

- the retained role of transitional FileSystem working stores is now fixed by
  `docs/adr/0017-transitional-filesystem-working-store-boundary-after-p2.md`
