# v1.0.14 Candidate Queue Design Note

Status: implementation_ready

## Goal

Define the first real resident app candidate queue implementation slice under ADR 0010.

This note turns Issue 3 into a concrete implementation start point.

## Decision

The first real resident app candidate queue should use:

```text
GET /app/screen-state/candidates
```

as its primary rendering payload.

`GET /app/candidates` remains the primary app-facing queue read model, but the first real queue
screen should consume the framework-neutral queue screen state builder rather than re-deriving queue
view state in the UI shell.

## Why this path

The current app layer already provides:

- `build_app_candidate_queue(items)`
- `build_candidate_queue_screen_state(queue_payload)`
- `GET /app/screen-state/candidates`

This state already expresses the queue screen in UI-oriented terms:

- `reviewable_count`
- `status_counts`
- `top_sections`
- `items`
- `default_sort`

The existing extension sidepanel also proves the expected queue semantics:

- active review context stays visible inside queue summary
- queue state highlights whether the active candidate is still present
- top sections are readable as compact summary rows
- queue items route directly into candidate detail

Using `GET /app/screen-state/candidates` for the first real queue screen keeps the Bridge-hosted
local web app shell focused on rendering and navigation wiring rather than duplicating queue summary
logic.

## Screen contents

The first real candidate queue should render:

1. queue summary
   - reviewable count
   - default sort
   - status counts
   - active review candidate presence when relevant
   - active review source and count when a review session exists
2. top sections summary
3. reviewable candidate list
4. detail navigation entry per candidate

## Required states

The first real queue screen must include:

- loading state
- auth failure state
- transport failure state
- empty-state handling when no reviewable candidates are present

## Boundary

This queue screen:

- does not show approved/rejected items in the default queue surface
- does not bypass current review semantics
- does not mutate candidate/profile state directly
- does not require bootstrap HTML for navigation

## Implementation shape

Recommended implementation order:

1. create a queue screen loader around `GET /app/screen-state/candidates`
2. render queue summary in stable field order
3. render top sections summary
4. render ordered queue items with detail navigation hooks
5. add loading/error/empty states
6. preserve active review context if present

## Relationship to other surfaces

- home screen should use `GET /app/screen-state/home`
- candidate detail should use `GET /app/screen-state/candidates/{id}`
- daemon panel should use `GET /app/screen-state/daemon`

This keeps the first real Sayane Resident App UI shell aligned with the existing screen-state contract instead of mixing
raw queue payload composition and derived UI state arbitrarily.
