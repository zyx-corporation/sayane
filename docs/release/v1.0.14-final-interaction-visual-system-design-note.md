# v1.0.14 Final Interaction and Visual System Design Note

Status: implemented

## Goal

Define the production interaction and visual-system direction for the first real resident app UI
shell.

This note turns Issue 10 into a concrete implementation start point.

## Implementation outcome

Issue 10 is now closed on the app-first resident shell path.

Delivered in the current slice:

- shared operator-workspace framing now wraps the resident shell with consistent navigation and session actions
- visible focus treatment, skip-link behavior, and bounded live-status annunciation are now explicit instead of incidental
- summary-first two-up panel rhythm is now shared across home, detail, and daemon shell surfaces
- narrow-width responsive behavior now keeps shell cards, forms, and tables usable without dropping core actions
- product language remains aligned with the existing i18n direction instead of renaming resident workflow terms for polish alone

## Decision

The production resident app should converge on one operator-facing interaction system across home,
queue, detail, action, capture, and daemon surfaces while preserving the current app-facing API and
screen-state boundaries.

The visual system should standardize:

- navigation hierarchy
- layout rhythm
- accessibility behavior
- responsive behavior
- stable product-language treatment for shared workflow terms

## Why this path

The current MVP work intentionally optimizes for boundary correctness first.

By the time Issue 10 starts, the remaining work is not new review semantics but sustained operator
usability:

- navigation should feel coherent across screens
- the UI should remain readable over longer sessions
- accessibility and responsive behavior should be explicit rather than incidental
- stable workflow concepts should not be renamed inconsistently across screens or locales

This keeps Issue 10 focused on operator-facing interaction quality rather than re-opening endpoint
or policy design.

## Interaction system direction

### Navigation

- one clear primary route structure for home, queue, detail, capture entry, and daemon panel
- predictable return paths between queue/detail and home/daemon

### Layout

- one shared panel/page rhythm
- readable summary-first hierarchy before raw payload detail

### Accessibility

- keyboard-first navigation expectations
- visible focus states
- bounded status/error annunciation

### Responsive behavior

- narrow-width layout remains usable without losing core review/daemon actions

### Product language

- preserve stable workflow terms where they act as product concepts
- keep wording aligned with the project’s i18n policy rather than renaming concepts only for visual polish

## Boundary

This issue:

- does not change app-facing endpoint contracts
- does not re-open candidate or daemon semantics
- does not replace semantic i18n keys with visual-only wording
- should preserve preview/observation wording on daemon surfaces

## Implementation shape

Recommended implementation order:

1. define shared navigation and page structure
2. define shared interaction states and focus behavior
3. define shared spacing/typography/status patterns
4. verify i18n-compatible product language treatment
5. add responsive/accessibility verification
