# Sayane 全体ロードマップと現在地点

## Phase 7 後回し・Phase 5以降 sayane-pro 並行開発方針

## 1. 現在の基本判断

Sayane は現在、Phase 3 の主要機能をほぼ完了し、Phase 4「Context Portability Export 実装」へ移行する直前にある。

現時点では、Obsidian / Vault 統合を急ぐ必要はない。Obsidian 統合は便利ではあるが、Sayane の正本管理、Candidate Review、Lineage、Import-to-Candidate の原則が十分に固まる前に導入すると、手編集されたメモ、生成された export、承認済み context、作業中ノートの境界が曖昧になる。そのため、Obsidian / Vault 統合は最後寄りの Phase に移動する。

また、Phase 5 以降では `sayane` と `sayane-pro` を並行開発する。Phase 4 までは OSS 側の中核機能を固める段階であり、Phase 5 以降は cross-LLM 実証、実験管理、評価レポート、組織利用、商用ダッシュボードなどが必要になるため、Pro 側の価値が立ち上がる。

基本方針は次の通りである。

```text
sayane:
  文脈を安全に扱うための中核基盤

sayane-pro:
  実験管理・高度可視化・組織運用・商用機能
```

## 2. Phase 0: 概念設計・思想整理

### 目的

Sayane の根本思想を定める段階である。

Sayane は、LLM の記憶を操作する道具ではない。Sayane は、ユーザーが自分の文脈を保持し、比較し、候補化し、評価し、承認し、必要に応じて他のLLMへ移送するための基盤である。

ここでの中心概念は次の通りである。

```text
Context Portability
Profile IR
Candidate Review
RDE
Lineage
Prompt Adapter
Local-first Context Store
Formation-layer Ethics
```

### 状態

完了済み。

### 主な成果

Sayane の基本思想、note / Zenn 連載構想、RDE との接続、Context Portability の初期構想、ADR 0001 の基礎が整理された。

## 6. Phase 4: Context Portability Export 実装

### 目的

Sayane の正本文脈を、他LLMへ安全に渡せる形式へ変換する段階である。

ここが次の主戦場である。

### 対象 Issue

```text
#141 Implement context export formats: yaml / markdown / prompt with scope and target adapters
```

### 必要機能

```text
export --format yaml
export --format markdown
export --format prompt
export --scope identity,interaction,writing,technical,...
export --target generic|chatgpt|claude|gemini|deepseek|openwebui
promptExport: never の除外
formation raw 情報の export guard
target adapter による整形
```

### 最初の目標

まずは Markdown export を安定させる。

```bash
sayane export --format markdown --scope identity,interaction,writing,technical --target chatgpt
```

Markdown は人間にもLLMにも検査しやすく、RDE評価に向いている。YAML は機械処理には向くが、そのままLLMに貼る形式としては最適ではない。

### OSS / Pro 関係

Phase 4 までは OSS 側 `sayane` を主軸にする。ここで作る export 基盤は Context Portability の中核であり、OSS に置くべきである。

## 7. Phase 5: Cross-LLM Transfer 実証

## 8. Phase 6: Import-to-Candidate Review

### 目的

他LLMや他環境から受け取った文脈バンドルを、直接 merge せず Candidate Review 経由で取り込む段階である。

### 対象 Issue

```text
#143 Design import-to-Candidate Review flow for portable context bundles
```

### 基本方針

```text
bundle import
→ parse / validate
→ Candidate生成
→ diff / RDE
→ approve / reject / edit
→ stored context
```

import は直接 Profile を書き換えない。必ず Candidate Review を通す。

### OSS 側 sayane の役割

```text
bundle import
Candidate生成
diff
approve / reject / edit
lineage
```

### sayane-pro 側の役割

```text
複数プロファイル管理
組織単位の import policy
承認ワークフロー
監査ログ閲覧
高度な conflict resolution
```

### 注意点

Phase 6 は Phase 5 の実験結果を見てから着手するのが望ましい。export 形式や adapter が安定しないうちに import を実装すると、受信側の仕様が揺れる。

### 目的


### OSS / Pro 境界


この境界により、Sayane の思想的正当性を OSS 側に残しつつ、事業化の余地を `sayane-pro` に確保する。

```text

```

この ADR では、Phase 5 以降における OSS / Pro の責務境界を明文化する。

## 10. Phase 8: Obsidian / Vault 統合

### 目的

Sayane の context を Obsidian vault や Git-backed knowledge base と接続する段階である。

### 位置づけ

この Phase は急がない。最後寄りに移動する。

### 理由

Obsidian 統合は便利だが、正本管理を壊すリスクがある。特に、Candidate Review を経由せずに手編集されたノートが正本に混ざると、Lineage と RDE の意味が崩れる。

### 統合時の原則

```text
canonical_context:
  Candidate Review 経由

working_notes:
  自由編集

exports:
  再生成可能

lineage:
  自動生成・監査用
```

Obsidian は「正本そのもの」ではなく、作業・閲覧・リンクの場として扱う。

## 14. RDE差異検証

### 保存された要素

Sayane の中心思想である「文脈はLLMの記憶ではなく、ユーザー側の正本として保持・監査・移送する」という設計思想は保存されている。

また、Context Portability を単なるファイル移動ではなく、意味変化の監査対象として扱う立場も維持されている。

### 変換された要素

初期構想では export/import や Obsidian 統合が比較的早い段階に置かれていたが、現在は Obsidian 統合を後回しにし、Phase 5 以降で `sayane-pro` と並行開発する構成に変換した。

これは、実装進捗と事業化可能性を踏まえた正当な変換である。

### 補完された要素

OSS / Pro 境界、Pro 側の商用機能、formation-layer consent UI、advanced lineage viewer、cross-LLM experiment dashboard などを補完した。

これにより、Sayane の思想的中核を OSS に残しつつ、事業収益化の導線を Pro 側に設ける構造が明確になった。

### 未解決の要素

RDE composite score の算出方法、target adapter ごとの最適プロンプト形式、import 時の conflict resolution、Pro 側の課金モデル、管理UIの範囲は未解決である。

### 逸脱リスク

Context Portability が「全部をLLMへ渡す」方向に滑るリスクがある。これは Sayane の設計思想に反する。

また、Pro 側に機能を寄せすぎると、OSS 側の中核思想が痩せるリスクがある。OSS には必ず Candidate Review、Profile IR、Lineage 基礎、export/import 基礎、formation-layer policy を残すべきである。

### 次回更新方針

次回は #141 の実装結果を踏まえ、Phase 4 の完了条件を具体化する。その後、#142 の ChatGPT baseline 実験を実施し、Context Portability の最初の実証データを取得する。

## 15. 結論

Sayane は現在、Phase 3 の主要部分を完了し、Phase 4 の Context Portability Export 実装へ進む地点にいる。

Phase 5 以降では `sayane-pro` を並行開発し、OSS 側には文脈管理の中核を残し、Pro 側には実験管理・高度監査・組織運用・商用可視化を置く。

Obsidian / Vault 統合は急がず、正本管理と Lineage の原則がさらに固まった後、最後寄りに進める。

次の一手は明確である。

```text
#141 を進める。
Markdown export を作る。
ChatGPT transfer baseline を実施する。
そこから Sayane は、構想ではなく実証へ入る。
```
