# v1.0.14 Production Auth Model Implementation Skeleton

Status: implementation_ready

## Goal

Turn the production auth-model design note into an implementation-ready skeleton for the first real
resident app UI shell.

This document assumes ADR 0010 and the production auth-model design note are already accepted.

## Target outcome

Replace the bootstrap bearer-plus-cookie convenience seam with:

- explicit bootstrap bearer verification
- dedicated local UI session issuance
- dedicated UI session use for follow-up browser activity

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. UI session model

Responsibility:

- define a dedicated local UI session artifact
- define issuance, validation, expiration/replacement, and invalidation behavior

Existing references:

- `src/sayane/bridge/auth.py`
- `src/sayane/bridge/routes/resident_app.py`

### 2. Bootstrap/session boundary

Responsibility:

- keep bearer auth as bootstrap/pairing entry
- stop using the static bearer as the steady-state browser credential

### 3. UI request wiring

Responsibility:

- update browser-driven follow-up reads/writes to rely on the dedicated UI session
- preserve current endpoint paths and app-facing payload contracts

### 4. Auth failure handling

Responsibility:

- render explicit auth failure in the UI shell
- distinguish invalid session behavior from generic transport failure where practical

### 5. Focused regression coverage

Responsibility:

- add focused tests for bootstrap entry, session continuation, invalid session, and session
  replacement

Existing references:

- `tests/test_bridge_api.py`

## Acceptance criteria

- the real UI no longer depends on the bootstrap-only cookie convenience seam
- the static Bridge bearer is no longer the normal long-lived browser session credential
- app-facing read/write boundaries remain unchanged
- auth failure behavior is visible and bounded
- focused auth/session regression coverage exists

## Test plan

### Contract-level

- verify app-facing endpoint paths and payload shapes remain stable
- verify bootstrap bearer still gates initial entry

### Session behavior

- bootstrap entry can mint or establish a dedicated UI session
- follow-up browser activity succeeds with that session
- invalid or expired session fails explicitly
- session replacement or invalidation behavior is test-covered

### UI behavior

- auth failure stays distinct from transport failure where practical
- follow-up navigation does not require re-supplying the raw bootstrap bearer each time

## Non-goals

- remote auth provider integration
- OS keychain storage
- daemon/candidate authorization redesign
- unrelated endpoint redesign

## Recommended execution order

1. define dedicated UI session model
2. separate bootstrap bearer from session issuance
3. rewire follow-up UI activity to the session
4. add explicit auth failure handling
5. add focused auth/session tests
