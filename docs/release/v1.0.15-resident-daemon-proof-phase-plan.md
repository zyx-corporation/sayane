# v1.0.15 Resident Daemon Proof Phase Plan

Status: implemented_for_identity_readiness_api_proof_cli_wording_line

## Goal

Turn the current resident-daemon preview and observation line into an explicit future proof phase
plan without overclaiming what the current resident app or daemon preview surfaces verify.

## Why this phase exists

The current resident app shell is complete enough for operator use on the local Bridge path, but it
still does not prove:

- daemon process identity
- daemon readiness
- API readiness

Those are intentionally deferred in the current app contract and release boundary.

## Scope

This phase should plan the smallest coherent implementation line that can introduce stronger daemon
evidence without collapsing the existing conservative boundary.

## Planned workstreams

### 1. Identity proof workstream

- define what production daemon identity proof actually requires
- separate planned-path preview from real process ownership evidence
- keep proof local, runtime-root scoped, and explicit about what is and is not verified

### 2. Readiness proof workstream

- define operation-class-specific readiness targets
- keep liveness distinct from readiness
- use conservative statuses for any unverified or ambiguous state

### 3. API readiness proof workstream

- define authenticated and authorized local API evidence
- keep socket presence distinct from usable API readiness
- make protocol/version compatibility explicit

### 4. Operator wording and UI boundary workstream

- keep the resident app daemon panel preview-oriented until stronger proof is implemented
- avoid silently upgrading current UI wording from preview to proof
- update operator docs only after actual evidence strength changes

## Exit criteria

This phase is ready to start implementation only when it can answer all of the following:

- what exact evidence proves identity
- what exact evidence proves readiness for one operation class
- what exact evidence proves API readiness
- what local-only safety properties constrain these checks
- what UI wording changes are justified by the new evidence

## Not in scope

- OS service installation or supervision UX
- tray or menu-bar supervision UI
- direct profile patch UI
- converting current preview wording into proof wording without new evidence

## Completion note

This proof phase has now been implemented through the local proof-contract and CLI proof-diagnostics
line:

- daemon identity proof contract
- daemon readiness proof contract
- daemon API readiness proof contract
- shared conservative proof status vocabulary
- aggregated proof-oriented CLI diagnostics
- operator/app wording alignment that preserves preview-oriented daemon-panel wording

## Recommended implementation order

1. daemon identity proof contract
2. daemon readiness proof contract
3. API readiness proof contract
4. conservative status vocabulary alignment
5. operator/UI wording update after evidence changes
