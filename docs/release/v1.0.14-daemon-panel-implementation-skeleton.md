# v1.0.14 Daemon Panel Implementation Skeleton

Status: implementation_ready

## Goal

Turn the daemon-panel design note into an implementation-ready skeleton for the first real resident
app UI shell.

This document assumes ADR 0010 and the daemon-panel design note are already accepted.

## Primary payload

Use:

```text
GET /app/screen-state/daemon
```

The daemon panel should render directly from `resident_app_daemon_panel_screen_state`.

## Expected payload sections

- `summary_cards`
- `next_actions`
- `runtime_init`
- `cleanup_preview`
- `repair_preview`

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. Daemon panel loader

Responsibility:

- fetch `GET /app/screen-state/daemon`
- own loading/error/retry behavior

Existing references:

- `extension/src/screen-state-reader.ts`
- `extension/src/bridge-client.ts`

### 2. Daemon panel state types

Responsibility:

- reuse the existing `DaemonPanelScreenState` shape
- avoid introducing a second daemon-specific view model if the existing shape is sufficient

Existing references:

- `extension/src/types.ts`
- `src/sayane/app/app_screen_states.py`
- `src/sayane/app/ui.py`

### 3. Summary cards renderer

Responsibility:

- render `summary_cards` in stable order
- preserve current keys:
  - `state`
  - `is_running_daemon`
  - `runtime_initialized`
  - `readiness_status`

### 4. Preview sections renderer

Responsibility:

- render `next_actions`
- render `runtime_init`
- render `cleanup_preview`
- render `repair_preview`
- keep preview sections readable without implying automatic action

### 5. Empty/error state handling

Responsibility:

- show explicit loading state
- show auth failure state
- show transport failure state
- show bounded empty state when a preview section has no entries

## Acceptance criteria

- the daemon panel renders from `GET /app/screen-state/daemon`
- the app does not need bootstrap HTML to show daemon overview preview
- summary cards and next actions are readable without opening raw payloads
- loading/auth/transport/empty states are visible and bounded
- wording remains preview/observation oriented

## Test plan

### Contract-level

- verify the UI reads `resident_app_daemon_panel_screen_state`
- verify summary card keys are consumed in stable order

### UI behavior

- loading state renders before fetch completion
- auth failure state renders on auth error
- transport failure state renders on fetch failure
- empty sections hide or render bounded empty state cleanly

### Navigation

- home quick link routes to daemon panel
- top daemon action routes to daemon panel

## Non-goals

- final design system
- production auth redesign
- direct daemon start/stop/restart mutation UI in the same issue
- proof-style daemon health certification

## Recommended execution order

1. wire daemon panel loader
2. render summary cards
3. render next actions
4. render preview sections
5. add loading/error/empty states
6. add focused UI tests
