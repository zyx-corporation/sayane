# v1.0.14 MVP UI State Handling and Regression Implementation Skeleton

Status: implementation_ready

## Goal

Turn the MVP UI state-handling design note into an implementation-ready skeleton for the first real
resident app UI shell.

This document assumes ADR 0010 and the state-handling design note are already accepted.

## Coverage target

This slice should cover the shared MVP behavior for:

- home
- queue
- detail/diff
- action flow
- clipboard capture
- daemon panel

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. Shared loader/error conventions

Responsibility:

- keep loading/error patterns consistent across screen loaders
- distinguish auth failure, transport failure, empty state, and operation error

Existing references:

- `extension/src/resident-app-sidepanel.ts`
- `extension/src/screen-state-reader.ts`

### 2. Safe fallback handling

Responsibility:

- preserve fail-closed queue fallback for detail refresh failure
- avoid rendering stale screen data after reload failures

Existing references:

- `extension/src/resident-app-sidepanel.ts`

### 3. Focused regression coverage

Responsibility:

- add focused checks for the main MVP path set
- prefer targeted contract/UI-path tests before broad E2E harness work

Existing references:

- `tests/test_bridge_api.py`
- `tests/test_acceptance_coverage.py`

### 4. Acceptance mapping updates

Responsibility:

- update acceptance-scenario mapping only where it helps track the MVP UI path set
- avoid creating purely ceremonial coverage entries

### 5. Localized feedback consistency

Responsibility:

- keep state and error feedback compatible with the current resident-app i18n direction
- avoid introducing final-only copy that diverges from existing wording policy

## Acceptance criteria

- each MVP screen has explicit loading/error expectations
- detail refresh failure falls back safely instead of leaving stale detail visible
- focused regression coverage exists for the main MVP path set
- loading/auth/transport/empty/operation-error handling is visible and bounded
- localized state/error wording remains compatible with the resident-app i18n path

## Test plan

### Contract-level

- verify screen-state contracts still support the expected MVP loaders
- verify mutation/capture paths still route through existing app-facing surfaces

### UI behavior

- loading state appears before screen fetch completion
- auth failure renders distinctly from transport failure
- bounded empty states render only for valid empty payloads
- detail refresh fallback returns to queue on reload failure

### Regression set

- one happy-path home load
- one queue -> detail path
- one detail -> diff read path
- one action-flow path
- one clipboard capture path
- one daemon panel path

## Non-goals

- full production GUI E2E suite
- production auth redesign
- broad visual regression infrastructure
- unrelated test cleanup

## Recommended execution order

1. align shared loader/error behavior
2. preserve safe fallback behavior
3. add focused regression checks
4. update acceptance mapping where useful
5. verify i18n-compatible state/error wording
