# v1.0.14 Real UI E2E and Regression Design Note

Status: implemented

## Goal

Define the production regression strategy for the real resident app UI.

This note turns Issue 12 into a concrete implementation start point.

## Implementation outcome

Issue 12 is now closed on the app-first resident shell path.

Delivered in the current slice:

- a GUI-level regression module now walks the main resident operator paths through the Bridge-hosted local shell
- the suite covers capture, queue/detail/diff, evaluate/revise/approve, reject, and daemon panel paths
- assertions stay anchored on stable route semantics, localized operator-visible shell copy, and app-facing review outcomes
- lower-level HTML, route, and contract-focused tests remain in place alongside the new path suite

## Decision

The production regression suite should expand from focused MVP checks into a GUI-level path set
centered on the real resident app while still anchoring on the existing app-facing contracts and
screen-state boundaries.

## Why this path

By Issue 12, manual walkthroughs are not enough, but the project still benefits from contract-first
discipline.

The real UI regression suite should therefore:

- prove the main operator paths end to end
- keep failures traceable back to stable app-facing contracts
- avoid coupling every test to incidental visual details

## Coverage set

The initial production regression suite should cover:

- capture
- queue
- detail
- diff
- evaluate / revise / approve / reject
- daemon panel

## Boundary

This issue:

- does not require full visual snapshot coverage for every surface
- should keep regression centered on stable operator paths and contracts
- does not replace lower-level focused tests

## Implementation shape

Recommended implementation order:

1. choose the main GUI path set
2. wire end-to-end harness entry/teardown
3. add path-level assertions anchored on stable screen semantics
4. keep lower-level contract tests alongside GUI checks
