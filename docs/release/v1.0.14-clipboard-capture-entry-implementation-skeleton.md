# v1.0.14 Clipboard Capture Entry Implementation Skeleton

Status: implementation_ready

## Goal

Turn the clipboard-capture entry design note into an implementation-ready skeleton for the first
real resident app UI shell.

This document assumes ADR 0010 and the clipboard-capture entry design note are already accepted.

## Primary write surface

Use:

```text
POST /app/capture-clipboard
```

The entry surface should create a candidate and route into the existing detail/queue flow.

## Request shape

Use the existing `AppCaptureClipboardRequest` fields:

- `content`
- `profile_id`
- `locale`

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. Capture submit client

Responsibility:

- send `POST /app/capture-clipboard`
- normalize auth/transport/validation errors

Existing references:

- `extension/src/bridge-client.ts`
- `src/sayane/bridge/routes/resident_app.py`

### 2. Capture entry UI

Responsibility:

- collect clipboard text or pasted clipboard text
- bind submit/busy/error state
- preserve locale/profile selection if exposed

### 3. Success navigation

Responsibility:

- route to the created candidate detail after success
- preserve compatibility with queue/home refresh behavior

### 4. Localized feedback

Responsibility:

- surface success and validation/error feedback through the current resident-app translation path
- keep wording consistent with the broader i18n policy for clipboard capture and review boundaries

### 5. Loading/error state handling

Responsibility:

- show capture-in-flight state
- show auth failure state
- show transport failure state
- show bounded validation feedback for empty/invalid clipboard input

## Acceptance criteria

- the real Sayane Resident App UI shell can create a pending candidate through `POST /app/capture-clipboard`
- successful capture routes to the created candidate detail
- returned candidate stays inside normal queue/detail/action flow
- loading/auth/transport/validation states are visible and bounded
- localized feedback remains compatible with the resident-app i18n path

## Test plan

### Contract-level

- verify the UI calls `POST /app/capture-clipboard` with the existing request shape
- verify created payload preserves clipboard capture metadata

### UI behavior

- capture success routes to the created candidate detail
- empty input fails cleanly
- auth failure renders explicitly
- transport failure renders explicitly

### Flow integration

- new candidate appears in queue after refresh
- created detail can proceed into the existing review flow

## Non-goals

- final design system
- production auth redesign
- selected-text capture in the same issue
- direct profile patch UI

## Recommended execution order

1. wire capture submit client
2. build capture entry UI
3. route success into candidate detail
4. add validation/error states
5. add localized feedback states
6. add focused UI tests
