# Sayane v1.0.12 Resident Daemon Policy Gate

Release date: 2026-06-18

## Summary

Sayane v1.0.12 packages the Resident Daemon Policy Gate work completed after v1.0.11.

This release does not implement a production daemon. Instead, it completes a broad set of architecture and governance policies required before actual resident daemon implementation may begin.

The release preserves the existing preview-only, non-mutating safety boundary.

## Added diagnostic command

This release includes the liveness evidence ceiling preview command added after the v1.0.11 PID file diagnostic work:

```bash
sayane app daemon-liveness-diagnostic --json
```

Runtime root override:

```bash
sayane app daemon-liveness-diagnostic --runtime-root /path/to/runtime --json
```

The command reports conservative liveness evidence states, such as:

```text
pid_missing_liveness_unverified
pid_unreadable_liveness_unverified
pid_empty_liveness_unverified
pid_invalid_liveness_unverified
pid_parsed_process_unverified
```

No current status claims daemon liveness.

## Policy gate now established

The resident daemon evidence ladder is now documented as:

```text
PID parse validity
-> process existence
-> process identity
-> daemon readiness
-> API readiness
```

The mutation and operation ladder is now documented as:

```text
diagnostic evidence
-> decision preview
-> operator authorization
-> mutation or control command
-> audit record
```

The implementation gate states that actual daemon implementation must not begin until a future implementation issue explicitly accepts the gate.

## Safety boundary retained

This release does not introduce:

- production resident daemon runtime
- daemon process start
- daemon process stop
- daemon process restart
- signal sending
- process supervision
- process probing
- PID file writes
- lock acquisition
- lock release
- lock stealing
- socket creation
- IPC endpoint exposure
- runtime directory creation
- stale artifact deletion
- artifact repair
- OS service integration
- persistent IPC credentials
- network-exposed resident API

## Recommended first implementation constraints

The implementation readiness gate recommends that the first actual resident daemon implementation should remain:

```text
local-only
single-user
no OS service integration
no network exposure
no cleanup apply
no repair apply
no lock stealing
no auto-start
no persistent IPC credentials
explicit operator start only
```

## RDE Delta-M Review

### Preserved

Resident daemon work remains preview-first.

Diagnostics remain non-mutating.

Actual daemon runtime implementation remains future work.

### Transformed

The resident daemon track has moved from isolated diagnostics into an explicit governance gate for future implementation.

### Supplemented

The release adds policy coverage for evidence, mutation, cleanup apply, repair, locks, sockets, runtime initialization, IPC authentication, operator consent, process control, OS service integration, and implementation readiness.

### Unresolved

The actual daemon state machine remains future work.

The actual IPC transport remains future work.

The runtime audit event schema remains future work.

### Deviation Risks

Do not imply that v1.0.12 implements a daemon.

Do not treat policy completeness as authorization to start processes or mutate filesystem state.

Do not allow preview commands to become mutating surfaces.
