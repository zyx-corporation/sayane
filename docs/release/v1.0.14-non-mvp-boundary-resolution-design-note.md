# v1.0.14 Non-MVP Boundary Resolution Design Note

Status: implemented

## Goal

Define the explicit production boundary decisions for the resident app surfaces that remain outside
MVP scope.

This note turns Issue 14 into a concrete implementation start point.

## Decision

The production release line should explicitly mark the following as either separately planned or
explicitly deferred:

- daemon identity proof
- daemon readiness proof
- OS service integration UI
- tray or background supervision UI
- direct profile patch UI

The important decision is not to solve them all in Issue 14, but to make the release boundary
auditable instead of leaving these topics implicit.

## Implemented resolution

| Topic | Resolution | Current release wording |
| --- | --- | --- |
| daemon identity proof | explicit defer | current resident app and daemon preview surfaces do not prove production daemon identity |
| daemon readiness proof | explicit defer | current resident app and daemon preview surfaces do not prove daemon readiness or API readiness |
| OS service integration UI | separate implementation plan | current release excludes service install/supervision UI |
| tray/background supervision UI | separate implementation plan | current release excludes tray, menu-bar, and background supervision UX |
| direct profile patch UI | explicit defer | current app-facing writes remain inside candidate review and do not patch profile state directly |

The resident app contract now carries the same boundary matrix through
`src/sayane/app/app_contract.py`, keeping the production exclusion line auditable from both docs and
the app-facing contract surface.

## Why this path

The current app-facing and bootstrap documents already repeat these boundaries, but they are spread
across multiple handoff and architecture files.

Issue 14 should consolidate them into one explicit production boundary resolution pass so the team
can answer:

- what is in production scope now
- what is deferred
- what needs a separate implementation plan

## Resolution categories

### Separate implementation plan

Use when the topic is a real future feature with enough value or risk to deserve its own tracked
workstream.

### Explicit defer

Use when the production release should clearly exclude the topic for now.

### Boundary reminder

Use when adjacent UI or operator docs must continue to warn against overclaiming current behavior.

## Boundary

This issue:

- does not imply the deferred topics are partially solved already
- does not silently convert preview/diagnostic signals into proof claims
- does not introduce direct profile patch capability through UI convenience

## Implementation shape

Recommended implementation order:

1. list each non-MVP topic explicitly
2. assign separate-plan or defer status
3. link each topic to the governing architecture/policy doc where available
4. update release/handoff docs so the final boundary stays visible
