# v1.0.14 Resident App Operator Handoff

Status: handoff_ready

## Supported startup path

The supported operator startup path for the current resident app is:

```bash
./scripts/run-macos-app-preview.sh
```

This launches the native macOS application against the local Bridge.

Useful variants:

```bash
./scripts/run-macos-app-preview.sh --foreground
./scripts/run-macos-app-preview.sh --no-build
```

The Bridge-hosted browser path remains available only as a debug-only compatibility surface.

If direct browser inspection is needed, start or reuse the local Bridge and bootstrap once through:

```bash
open -a "Google Chrome" "http://127.0.0.1:38741/app/ui?bootstrap_token=$(cat ~/.sayane/bridge.token)"
```

After that first hop, the browser can continue to use:

```text
http://127.0.0.1:38741/app/ui
```

Do not use a separate static-site URL such as `http://127.0.0.1:8008/index.html`.

## Packaging and runtime expectations

- current packaging model is local Python CLI + Local Bridge + native macOS app shell
- packaging-model decision candidates now exist in contract form, but only the current CLI-first line is supported
- current host/runtime boundary is `127.0.0.1:38741`
- native app is the primary operator-facing shell over existing app-facing resident endpoints
- browser access is local-only and remains debug / smoke / handoff / fallback only

## Config and local data

Current operator-relevant local paths:

```text
~/.sayane/bridge.token             bootstrap bearer credential
~/.sayane/bridge.ui-session.json   dedicated local UI session artifact
~/.sayane/profiles/default/        default profile root
~/.sayane/candidates/              captured candidates before approval
~/.sayane/run/                     resident daemon runtime-root preview area
```

Current assumptions:

- the initial `/app/ui` bootstrap still depends on `~/.sayane/bridge.token`
- follow-up local UI activity depends on the dedicated UI session cookie minted by the Bridge
- deleting or invalidating the UI session is expected to require a fresh `/app/ui` bootstrap entry
- candidate writes still flow through review endpoints and do not patch profile state directly

## First operator checks

After startup, the first checks should be:

```bash
TOKEN=$(cat ~/.sayane/bridge.token)
curl -s http://127.0.0.1:38741/health
curl -s -H "Authorization: Bearer $TOKEN" http://127.0.0.1:38741/app/contract
curl -s -H "Authorization: Bearer $TOKEN" http://127.0.0.1:38741/app/overview
```

Then in the native app:

1. confirm the Home screen renders
2. confirm Queue / Detail / Daemon navigation works
3. confirm Bridge status shows reconnect / startup guidance when applicable
4. export or copy handoff material from the native app when needed

Only if browser-local compatibility needs checking:

1. open `/app/ui?bootstrap_token=...` once
2. confirm the home screen renders
3. confirm queue/detail/daemon navigation works
4. use the shell logout action when the browser session should be invalidated

If Chrome is available on macOS, prefer the bootstrap URL above so the first UI session is created
in the browser you are actually testing.

## Common troubleshooting

### `/app/ui` returns `401`

Check:

- `~/.sayane/bridge.token` exists
- the browser/bootstrap request was started through the current Bridge host/port
- the Bridge process was restarted and the old token was not copied from another home

### follow-up shell requests return `Missing or invalid resident app UI session`

Check:

- `/app/ui` was opened successfully in the current browser session
- the UI session was not explicitly ended with logout
- the Bridge process was not restarted after the session cookie was minted

Recovery:

1. reopen `/app/ui`
2. if needed, restart `sayane serve`
3. bootstrap again from the new local shell session

### browser opens but queue/detail actions seem stale

Check:

- refresh the visible shell state
- confirm candidate status through `/app/candidates` or `/app/ui-state/candidates`
- confirm the action result via the redirect notice or JSON action response

### daemon panel looks degraded or incomplete

Check:

- `sayane app daemon-overview --json`
- `sayane app daemon-operator-phase-status`
- `sayane app daemon-packaging-status --json`
- `sayane app daemon-packaging-status`
- `sayane app daemon-service-control-boundary --json`
- `sayane app daemon-service-control-boundary`
- `sayane app daemon-supervision-status --json`
- `sayane app daemon-supervision-status`
- `sayane app daemon-recovery-consent-status --json`
- `sayane app daemon-recovery-consent-status`
- `sayane app daemon-service-targets-status`
- `sayane app daemon-readiness-diagnostic --operation-class bridge_health --json`
- `sayane app daemon-proof-diagnostics --operation-class bridge_health --json`
- runtime preview commands referenced by the daemon panel

Do not interpret degraded or unverified daemon output as proof failure for a production daemon.

### recovery or consent path needs clarification

Check:

- `sayane app daemon-recovery-consent-status --json`
- `sayane app daemon-recovery-consent-status`
- `sayane app daemon-service-control-boundary --json`
- `sayane app daemon-service-control-boundary`
- `sayane app daemon-supervision-status --json`
- `sayane app daemon-supervision-status`
- `sayane app daemon-proof-diagnostics --operation-class bridge_health --json`

Do not turn recovery guidance into mutation without the current CLI confirmation path.

## Post-app operator read surfaces

The current post-app operator line now has three aligned read layers:

1. aggregated JSON payloads for machine-readable review
2. framework-neutral daemon screen state plus app-shell / legacy-daemon-panel read-only rendering
3. human-readable CLI summaries for packaging, service targets, service/control boundary, supervision, recovery/consent, and aggregate operator phase status

These layers are intentionally aligned around the same conservative contract. They do not imply
OS-service activation, background supervision shipment, or proof-style daemon claims.
