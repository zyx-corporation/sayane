# v1.0.14 Clipboard Capture Entry Design Note

Status: implementation_ready

## Goal

Define the first real resident app clipboard-capture entry implementation slice under ADR 0010.

This note turns Issue 6 into a concrete implementation start point.

## Decision

The first real resident app candidate-creation entry should use:

```text
POST /app/capture-clipboard
```

as its primary write surface.

The real Sayane Resident App UI shell should treat clipboard capture as a candidate-creation action that always enters
the existing review boundary and routes the created candidate into queue/detail flow.

## Why this path

The current resident app boundary already provides:

- app contract write-surface declaration for `POST /app/capture-clipboard`
- JSON app-facing clipboard capture returning `CandidateUpdate`
- bootstrap HTML form route over the same underlying runtime service
- a narrow `AppCaptureClipboardRequest` shape of `content`, `profile_id`, and `locale`

The current implementation already proves key capture semantics:

- clipboard text creates a pending candidate rather than writing profile state directly
- returned payload marks `capture_surface` as `resident_app_bridge`
- created candidates enter the same candidate queue/detail/diff flow as other reviewable items
- locale can travel with capture and should stay compatible with the current resident-app i18n seam

Using `POST /app/capture-clipboard` for the first real candidate-creation entry keeps the
Bridge-hosted Sayane Resident App local shell aligned with the existing service boundary rather than inventing a
separate creation path.

## Entry semantics

### Input

- sends explicit clipboard text
- may include `profile_id`
- may include `locale`

### Output

- returns `CandidateUpdate`
- identifies the created candidate id
- keeps `source.type == "clipboard"`
- keeps `capture_meta.capture_source == "clipboard"`

### Navigation

- should route to the created candidate detail after success
- should keep queue/home refreshable after candidate creation

## Required states

The first real capture entry must include:

- submit-in-flight state
- auth failure state
- transport failure state
- validation failure state for empty/invalid input
- success navigation into the created candidate

## Boundary

This capture-entry slice:

- does not write profile state directly
- does not bypass candidate review semantics
- does not collapse clipboard preflight/preview wording into misleading final-commit language
- should keep user-facing copy compatible with the current i18n direction instead of hard-coding a final-only phrase set

## Implementation shape

Recommended implementation order:

1. add clipboard capture submit helper for `POST /app/capture-clipboard`
2. bind capture input and submit state in the chosen UI shell
3. route success into the created candidate detail
4. refresh queue/home state after navigation when needed
5. surface validation/auth/transport failures
6. wire localized feedback through the current resident-app translation path

## Relationship to other surfaces

- home screen is the natural launch surface for first capture entry
- created candidates should flow directly into candidate detail
- queue should show the new pending candidate on refresh
- action flow stays separate and begins after candidate creation

This keeps the first real Sayane Resident App UI shell aligned with the existing candidate creation boundary and with
the project’s clipboard-capture-first direction.
