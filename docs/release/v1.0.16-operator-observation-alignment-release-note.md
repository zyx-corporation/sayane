# Sayane v1.0.16 — Operator Observation Alignment

## Summary

Sayane `v1.0.16` does not close the operator packaging and supervision phase.

Instead, it aligns the current **observation layer** across:

- CLI read surfaces
- Bridge-hosted resident app shell
- macOS native app preview
- operator handoff and getting-started documents

This keeps the next operator-facing release line auditable without silently turning observation
surfaces into control commitments.

## Highlights

- bridge-hosted daemon shell now exposes proof-oriented diagnostic commands alongside operator-phase data
- native macOS app runbook and bridge-hosted shell now present the same proof-diagnostic read path
- getting-started and operator handoff now point to operator-phase and proof-diagnostic reads explicitly
- current packaging / service / supervision / recovery line remains read-only and preview-first

## Boundary

- this release does not finalize the supported packaging model
- this release does not ship OS service install / enable / disable UI
- this release does not ship tray or background supervision UX
- this release does not widen daemon proof claims beyond the current proof-oriented diagnostic line

## Recommended reads

```bash
sayane app daemon-packaging-status --json
sayane app daemon-service-control-boundary --json
sayane app daemon-supervision-status --json
sayane app daemon-recovery-consent-status --json
sayane app daemon-operator-phase-status --json
sayane app daemon-proof-diagnostics --operation-class bridge_health --json
```

## Why this release exists

The operator packaging and supervision phase is still open, but operators already need one aligned
review path across browser shell, native preview, CLI, and handoff docs. This release closes that
alignment gap while keeping the unresolved decisions explicit.
