# v1.0.14 UI Implementation Task Breakdown

Status: implemented_on_bridge_hosted_local_shell

## Goal

Translate the current resident app surfaces into the first actual UI implementation slice.

## Implemented outcome

The Bridge-hosted Sayane Resident App local shell now completes Tasks 1-7 on the current release
line:

- bootstrap wiring is implemented through `GET /app/ui`
- home summary is implemented through `GET /app/ui` plus `GET /app/ui-state/home`
- candidate queue/detail/diff are implemented through `ui-state` JSON surfaces and local HTML companions
- candidate evaluate/revise/approve/reject are implemented through `ui-action` JSON writes and local HTML form companions
- clipboard capture entry is implemented through app-facing and local-shell write surfaces
- daemon panel is implemented through `GET /app/ui-state/daemon` and `GET /app/ui/daemon`

Follow-on completion issues 8-14 are now also closed on the same release line.

## Recommended order

### Task 1: Bootstrap wiring

- fetch `GET /app/contract`
- fetch `GET /app/overview`
- render failure state for auth / transport errors

Current baseline:

- implemented as local Bridge HTML bootstrap at `GET /app/ui`
- the page is server-rendered from contract + overview payloads

Done when:

- the app can start from one primary payload
- the preferred entrypoint is not hardcoded outside the contract

### Task 2: Home summary screen

- render overview summary cards
- render top review items
- render top daemon next actions

Current baseline:

- implemented in `GET /app/ui`
- summary cards / top review items / daemon next actions are visible without extra fetch wiring

Done when:

- a user can understand current app state without opening deeper panels

### Task 3: Candidate queue screen

- fetch `GET /app/candidates`
- render reviewable candidate list
- navigate to detail screen

Current baseline:

- implemented as local Bridge HTML queue at `GET /app/ui/candidates`
- queue rows link into local detail / diff HTML screens

Done when:

- pending/evaluated candidates are readable in one place

### Task 4: Candidate detail and diff

- fetch `GET /app/candidates/{id}`
- fetch `GET /app/candidates/{id}/diff`
- render proposal, evaluation, and diff

Current baseline:

- implemented as local Bridge HTML detail / diff screens
- current rendering is intentionally minimal and presentation-only

Done when:

- the user can inspect what the candidate is and what approval would change

### Task 5: Candidate actions

- wire `POST /app/candidates/{id}/evaluate`
- wire `POST /app/candidates/{id}/revise`
- wire `POST /app/candidates/{id}/approve`
- wire `POST /app/candidates/{id}/reject`

Current baseline:

- local HTML detail screen now submits evaluate / revise / approve / reject forms
- action handlers redirect back into queue/detail HTML flow

Done when:

- the review flow can complete without leaving the app surface

### Task 6: Clipboard capture entry

- wire `POST /app/capture-clipboard`
- route created candidate into queue/detail flow

Current baseline:

- local HTML home screen now submits clipboard capture form
- created candidate redirects directly into local detail screen

Done when:

- clipboard capture becomes the first app-native candidate creation path

### Task 7: Daemon panel

- fetch `GET /app/daemon-overview`
- render lifecycle/readiness/next-actions panel

Current baseline:

- implemented as local Bridge HTML daemon panel at `GET /app/ui/daemon`
- renders state summary, readiness summary, next actions, runtime-init, cleanup, and repair previews

Done when:

- the app can show local daemon state without claiming production daemon proof

## Guardrails

- keep the first UI read path centered on `GET /app/overview`
- avoid adding profile patch endpoints to unblock UI work
- prefer summary-first rendering, detail-on-demand fetches second
- preserve candidate lineage by revising into new pending candidates

## Explicit non-goals

- final visual design system
- tray integration
- background daemon supervision UI
- direct profile editor
- production auth hardening

## Current auth note

The current bootstrap HTML uses:

- bearer token on the first `GET /app/ui`
- local UI session cookie for follow-up HTML navigation and form posts

This is a bootstrap convenience seam only. It is not the final production auth design.

## Current interaction note

The current bootstrap HTML also includes:

- shared top-level navigation across home / candidate / daemon screens
- redirect-based success notices after form actions
- redirect-based error feedback for invalid local action flows
- queue-level evaluation summary visibility
- queue-level pending/evaluated summary counts
- detail-level action guidance before approval

## Current follow-on line

What remains after this task set is intentionally outside the core app-completion slice:

- daemon identity proof
- daemon readiness / API readiness proof
- OS service integration UI
- tray/background supervision UI
- direct profile patch UI

These are tracked as explicit defer or separate-plan items in the current app contract and release
notes.
