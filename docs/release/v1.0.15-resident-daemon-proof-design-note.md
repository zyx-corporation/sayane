# v1.0.15 Resident Daemon Proof Design Note

Status: implemented_for_issue_1_2_3_4_5_6_identity_readiness_api_status_cli_wording_contracts

## Goal

Define the first coherent implementation direction for resident daemon proof work after the resident
app local-shell completion line.

## Decision

The daemon proof phase should progress in strict evidence order:

1. process identity proof
2. daemon readiness proof
3. API readiness proof

The phase must not collapse these into one generic "ready" or "running safely" claim.

## Why this path

Current Sayane surfaces already expose:

- daemon preview
- liveness observation
- readiness-diagnostic preview
- operator-facing conservative wording

What is missing is not raw visibility, but stronger evidence contracts.

The next phase should therefore define proof contracts before it changes operator-facing wording or
app-shell claims.

## Proof direction

### Identity proof

Identity proof should answer:

- which process is being evaluated
- why it is considered a Sayane daemon
- which runtime root that proof applies to
- what ambiguity remains, if any

### Readiness proof

Readiness proof should answer:

- which operation class is being checked
- which prerequisites were verified
- whether the daemon is ready for that operation class now
- which conditions downgrade the result to unverified, degraded, or not ready

### API readiness proof

API readiness proof should answer:

- whether the API surface is reachable
- whether it is authenticated and authorized
- whether protocol and version expectations match
- whether the requested operation class is actually usable

## Boundary

This phase:

- does not equate liveness with readiness
- does not equate socket presence with API readiness
- does not upgrade preview UI wording to proof wording before new evidence exists
- does not introduce OS service or supervision UX

## First implementation target

The first implementation target should be a proof-oriented contract and status vocabulary that can
be consumed consistently by:

- CLI diagnostics
- future operator runbooks
- app-shell daemon presentation

The phase should begin there instead of starting from UI polish or process-control actions.

## Implemented Issue 1 outcome

The first proof-oriented identity contract now exists as a conservative preview line:

- app-layer identity-proof payload
- CLI read surface
- explicit `identity_not_running` / `identity_unverified` / `manual_review_required` statuses

It still does not claim verified daemon identity, but it gives the proof phase one stable contract
to build on before readiness and API readiness work.

## Implemented Issue 2 outcome

The first proof-oriented readiness contract now also exists as a conservative preview line:

- operation-class-aware readiness-proof payload
- CLI read surface
- explicit `readiness_not_ready` / `readiness_unverified` / `readiness_degraded` / `manual_review_required` statuses

It still does not claim API readiness, but it separates readiness-proof evaluation from the older
combined readiness/API diagnostic preview.

## Implemented Issue 3 outcome

The first proof-oriented API-readiness contract now also exists as a conservative preview line:

- API-readiness-proof payload
- CLI read surface
- explicit `api_unreachable` / `api_unauthenticated` / `manual_review_required` statuses
- explicit observed reachability/authentication/authorization fields

It still does not claim verified API usability, but it separates API proof evaluation from both the
identity-proof and readiness-proof contracts.

## Implemented Issue 4 outcome

The proof contracts now share one conservative status vocabulary module plus explicit downgrade
reasons:

- shared proof status enum across identity/readiness/API proof contracts
- shared downgrade-reason enum across identity/readiness/API proof contracts
- explicit `downgrade_reason` fields in all three proof payloads

This keeps ambiguity and downgrade behavior visible without introducing any plain generic `ready`
status.

## Implemented Issue 5 outcome

The proof contracts are now exposed through both focused CLI reads and one aggregated diagnostics
surface:

- `sayane app daemon-identity-proof`
- `sayane app daemon-readiness-proof`
- `sayane app daemon-api-readiness-proof`
- `sayane app daemon-proof-diagnostics`

The aggregated command keeps the proof targets separate while letting operators inspect the current
proof line in one non-mutating call.

## Implemented Issue 6 outcome

Operator-facing and app-shell wording now stays aligned with the current proof line:

- resident app daemon panel remains explicitly preview-oriented
- operator handoff now points to proof-oriented CLI diagnostics without overclaiming verified proof
- CLI and docs distinguish preview diagnostics from proof-oriented preview diagnostics

## Recommended implementation order

1. define identity-proof payload contract
2. define readiness-proof payload contract
3. define API-readiness payload contract
4. align conservative status vocabulary across those contracts
5. update operator/app wording only after the above exist
