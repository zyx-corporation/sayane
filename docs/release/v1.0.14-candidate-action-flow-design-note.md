# v1.0.14 Candidate Action Flow Design Note

Status: implementation_ready

## Goal

Define the first real resident app candidate action-flow implementation slice under ADR 0010.

This note turns Issue 5 into a concrete implementation start point.

## Decision

The first real resident app action flow should mutate through the existing app-facing write surfaces:

```text
POST /app/candidates/{id}/evaluate
POST /app/candidates/{id}/revise
POST /app/candidates/{id}/approve
POST /app/candidates/{id}/reject
```

The real Sayane Resident App UI shell should treat `CandidateDetailScreenState.allowed_actions` as the primary action
availability contract and should not re-infer approval/rejection readiness on its own.

## Why this path

The current resident app boundary already provides:

- app contract write-surface declarations for evaluate/revise/approve/reject
- JSON app-facing mutation endpoints returning `CandidateUpdate`
- bootstrap HTML form routes over the same underlying candidate lifecycle
- detail screen state that exposes `allowed_actions`

The existing implementation already proves key action semantics:

- evaluate promotes review state without directly mutating profile state
- revise creates a new pending candidate rather than rewriting approved history in place
- approve can require `force_critical`, `override_reason`, and explicit confirmation for specific sections
- reject stays inside the current review boundary

Using the existing app-facing POST surfaces for the first real action flow keeps the Bridge-hosted
local web app shell aligned with current candidate lifecycle behavior rather than inventing a second
mutation contract.

## Action semantics

### Evaluate

- uses `POST /app/candidates/{id}/evaluate`
- accepts level `1..3`
- should refresh detail/queue state after success

### Revise

- uses `POST /app/candidates/{id}/revise`
- creates a new pending candidate
- should route the user to the new candidate detail after success

### Approve

- uses `POST /app/candidates/{id}/approve`
- must preserve current critical-approval safeguards
- must surface explicit confirmation requirements rather than bypassing them in the UI

### Reject

- uses `POST /app/candidates/{id}/reject`
- keeps rejection inside the existing candidate review flow
- should return the user to a refreshed queue/detail state without profile patch shortcuts

## Required states

The first real action flow must include:

- submit-in-flight state per action
- auth failure state
- transport failure state
- operation error handling using server-returned error payloads
- success refresh/navigation after mutation

## Boundary

This action-flow slice:

- does not add new mutation endpoints
- does not bypass `allowed_actions`
- does not bypass critical-approval or explicit-confirmation safeguards
- does not introduce direct profile patch UI
- must keep user-facing copy compatible with the existing i18n direction rather than hard-coding final-only strings

## Implementation shape

Recommended implementation order:

1. add action submit helpers for evaluate/revise/approve/reject
2. bind helpers from candidate detail state using `allowed_actions`
3. refresh detail/queue state after success
4. route revised candidates into the new candidate detail
5. surface operation errors from the existing Bridge payloads
6. add localized success/error copy through the current resident-app i18n path

## Relationship to other surfaces

- candidate detail should remain the primary action-launch surface
- candidate queue should reflect post-mutation state after refresh
- clipboard capture remains a separate candidate-creation entry flow
- daemon panel remains outside the candidate mutation path

This keeps the first real Sayane Resident App UI shell aligned with the existing review lifecycle and with the current
Bridge/UI localization seam.
