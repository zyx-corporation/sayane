Status: implementation_ready

## Goal

Turn the resident daemon proof design note into an implementation-ready skeleton.

## Expected file touch areas

### 1. App-layer proof contracts

- daemon identity proof payload definition
- daemon readiness proof payload definition
- API readiness proof payload definition
- shared conservative status vocabulary

### 2. CLI proof surfaces

- proof-oriented diagnostic command surfaces
- operation-class-specific invocation shape
- proof status and ambiguity reporting

### 3. Operator/app wording

- operator runbook updates after evidence changes
- daemon panel wording updates only where justified
- explicit distinction between preview and proof surfaces

### 4. Focused verification

- unit coverage for proof-status classification
- CLI coverage for proof payload shape

- identity, readiness, and API readiness are separate proof targets
- conservative status vocabulary is shared and explicit
- operator/app wording does not overclaim the implemented evidence
- new proof contracts are usable without introducing service/supervision UX
