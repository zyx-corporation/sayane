# v1.0.14 Candidate Detail and Diff Implementation Skeleton

Status: implementation_ready

## Goal

Turn the candidate detail/diff design note into an implementation-ready skeleton for the first real
resident app UI shell.

This document assumes ADR 0010 and the candidate detail/diff design note are already accepted.

## Primary payload

Use:

```text
GET /app/screen-state/candidates/{id}
```

The detail screen should render directly from `resident_app_candidate_detail_screen_state`.

## Companion payloads

Use these bounded companion reads after the detail state resolves:

- `GET /app/candidates/{id}/diff`
- candidate lineage read path already used by the current resident-app detail surface

## Expected payload sections

- `ui_summary`
- `allowed_actions`
- `proposal`
- `evaluation`
- `content`
- `diff_available`

## Expected file touch areas

The exact filenames depend on the chosen Bridge-hosted Sayane Resident App local shell, but the work should map
to these responsibilities:

### 1. Detail screen loader

Responsibility:

- fetch `GET /app/screen-state/candidates/{id}`
- own loading/error/fallback behavior

Existing references:

- `extension/src/screen-state-reader.ts`
- `extension/src/bridge-client.ts`

### 2. Detail screen state types

Responsibility:

- reuse the existing `CandidateDetailScreenState` shape
- avoid introducing a second detail-specific view model if the existing shape is sufficient

Existing references:

- `extension/src/types.ts`
- `src/sayane/app/app_screen_states.py`
- `src/sayane/app/app_candidate_views.py`

### 3. Detail summary renderer

Responsibility:

- render candidate id and active review context
- render `diff_available`
- render `ui_summary` in stable order

### 4. Proposal/evaluation/content sections

Responsibility:

- render `proposal`
- render `evaluation`
- render `content`
- preserve readable empty-state handling when content is absent

### 5. Action availability and review aids

Responsibility:

- render `allowed_actions`
- keep action hints bounded to current server-provided semantics
- render diff preview only when `diff_available` is true
- render lineage preview without inventing new mutation affordances

### 6. Fallback/error state handling

Responsibility:

- show explicit loading state
- show auth failure state
- show transport failure state
- fail closed back to queue when the detail candidate can no longer load

## Acceptance criteria

- the detail screen renders from `GET /app/screen-state/candidates/{id}`
- the app does not need bootstrap HTML to show detail/diff
- detail summary is readable without opening raw payloads
- diff preview appears only when available
- loading/auth/transport/fallback states are visible and bounded
- queue return and active-review return navigation remain obvious

## Test plan

### Contract-level

- verify the UI reads `resident_app_candidate_detail_screen_state`
- verify detail summary fields are consumed in stable order

### UI behavior

- loading state renders before fetch completion
- auth failure state renders on auth error
- transport failure state renders on fetch failure
- queue fallback occurs when the visible detail candidate can no longer load
- diff section hides cleanly when `diff_available` is false

### Navigation

- queue item routes into detail
- queue return routes back to queue
- active review shortcut routes to current active candidate when present

## Non-goals

- final design system
- production auth redesign
- evaluate/revise/approve/reject mutation UI in the same issue
- direct profile patch UI

## Recommended execution order

1. wire detail screen loader
2. render detail summary
3. render proposal/evaluation/content
4. render action hints
5. add diff preview and lineage preview
6. add loading/error/fallback states
7. add focused UI tests
